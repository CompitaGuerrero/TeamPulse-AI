# AGENTS.md

# PRD и правила для AI coding agent

## Проект: TeamPulse AI

Мини-дашборд для команды, который помогает быстро получать структурированные ответы через AI.

---

## 1. Роль агента

Ты — AI coding agent, который должен быстро создать рабочий веб-сервис по конкурсному заданию.

Твоя задача — не просто написать код, а собрать законченный MVP:

- чистая структура проекта;
- понятный UI;
- реальная LLM-интеграция;
- специализированный system prompt;
- обработка ошибок;
- история запросов;
- копирование ответа;
- аккуратная презентационная логика.

Работай как senior frontend/fullstack developer с хорошим UX-мышлением.

---

## 2. Контекст задачи

Нужно создать веб-сервис — мини-дашборд, который помогает команде быстро получать информацию через AI.

Сервис должен выглядеть не как обычный чат, а как небольшой продукт для команды: рабочий AI-помощник, который помогает быстро принимать решения, разбирать вопросы, видеть риски и следующие шаги.

---

## 3. Главная цель продукта

Создать мини-дашборд **TeamPulse AI**.

Пользователь вводит рабочий вопрос, нажимает кнопку **«Спросить AI»**, получает структурированный ответ в красивом виде, может скопировать ответ и видеть историю последних 5 запросов.

Продукт должен демонстрировать:

- работоспособность;
- чистую структуру проекта;
- аккуратный UI/UX;
- реальную LLM-интеграцию;
- специализированный AI-промпт;
- обработку ошибок;
- готовность к демонстрации в 3–5 минутном видео.

---

## 4. Что хочет заказчик

Заказчик хочет проверить не просто способность вызвать API, а способность быстро собрать маленький, но осмысленный AI-продукт.

Важно показать:

1. умение быстро доводить интерфейс до рабочего состояния;
2. понимание, как встроить LLM не формально, а через роль, промпт и обработку ошибок;
3. UX-мышление;
4. чистую структуру проекта;
5. способность объяснить принятые решения;
6. умение работать в формате быстрого MVP.

Главная ошибка — сделать обычный ChatGPT-клон без специализации.

---

## 5. Критерии оценки конкурса

При разработке учитывай систему оценки:

| Критерий | Вес |
|---|---:|
| Работает ли сервис | 25 |
| Качество UI/UX | 25 |
| Чистота кода | 20 |
| Презентация и объяснение решений | 20 |
| Креативность | 10 |
| Реальная LLM с настроенными промптами | +10 |

Главный приоритет: рабочий, аккуратный и законченный MVP.

---

## 6. Что считается неправильным решением

Не делай так:

- просто input + button + сырой ответ от API;
- обычный ChatGPT-клон без специализации;
- отсутствие system prompt;
- отсутствие обработки ошибок;
- падение приложения при ошибке API;
- хранение API-ключа на клиенте;
- сырой JSON на экране;
- неаккуратный UI без состояний загрузки, пустых состояний и обратной связи;
- хаотичная структура файлов;
- слишком сложная архитектура, которую невозможно закончить быстро.

## 7. Рекомендуемый стек

Если проект ещё не создан, используй:

- Next.js;
- TypeScript;
- Tailwind CSS;
- OpenAI API или совместимый LLM API;
- server route для обращения к LLM;
- localStorage для истории последних 5 запросов.

Если проект уже создан на другом стеке, адаптируйся к текущей структуре, но сохрани все функциональные требования.

## 8. Обязательный функционал

### 8.1 Поле ввода вопроса

Нужно реализовать:

- `textarea` для вопроса;
- placeholder с примером рабочего вопроса;
- валидацию пустого вопроса;
- disabled-состояние во время запроса;
- отправку через кнопку;
- желательно: отправку через `Cmd/Ctrl + Enter`.

Пример placeholder:

```txt
Например: Как быстро подготовить запуск лендинга к понедельнику?
```

---

### 8.2 Кнопка «Спросить AI»

Требования:

- текст: **Спросить AI**;
- во время загрузки: **AI думает...** или аналогичный текст;
- disabled, если поле пустое или идёт запрос;
- визуально должна быть основным CTA.

---

### 8.3 Ответ AI

Ответ должен отображаться в красивой карточке.

Обязательные требования:

- нормальная типографика;
- визуальное разделение блоков;
- поддержка переносов строк;
- аккуратное отображение списков;
- не показывать сырой JSON;
- если ответа ещё нет — показать empty state.

Структура ответа должна быть управляемой через system prompt.

---

### 8.4 История последних 5 запросов

Нужно реализовать:

- хранение последних 5 запросов;
- сохранение истории между перезагрузками страницы через `localStorage`;
- последние запросы сверху;
- клик по элементу истории должен подставлять вопрос обратно в поле ввода или повторно открывать связанный ответ, если это удобно;
- если истории нет — показать empty state.

Минимальное требование: отображать последние 5 вопросов.

Лучшее решение: хранить не только вопрос, но и связанный с ним ответ.

---

### 8.5 Кнопка «Копировать ответ»

Требования:

- копировать последний ответ AI в буфер обмена;
- использовать Clipboard API;
- после копирования показать обратную связь: **Скопировано**;
- если ответа нет — кнопка disabled или скрыта;
- обработать ошибку копирования.

---

### 8.6 Обработка ошибок

Обязательно обработать:

- пустой вопрос;
- отсутствие API-ключа;
- ошибка API;
- ошибка сети;
- слишком долгий ответ;
- неизвестная ошибка.

Пользователь не должен видеть crash приложения.

Ошибки показывать человеческим языком в UI.

Пример:

```txt
Не удалось получить ответ от AI. Проверьте API-ключ или попробуйте позже.
```

## 9. LLM-интеграция

### 9.1 Главное требование

Нужна реальная LLM-интеграция с настроенным system prompt.

Просто вызвать API и вывести ответ нельзя. AI должен иметь специализацию.

---

### 9.2 Роль AI-ассистента

AI должен работать как командный помощник для быстрого анализа рабочих вопросов.

Он должен помогать команде:

- быстро разобраться в вопросе;
- получить краткий вывод;
- увидеть возможные риски;
- получить следующие действия;
- понять, какие данные нужно уточнить.

---

### 9.3 System prompt

Используй следующий system prompt или его аккуратно адаптированную версию:

```txt
Ты — TeamPulse AI, внутренний AI-ассистент команды.

Твоя задача — помогать команде быстро получать структурированные ответы по рабочим вопросам, проектам, решениям, рискам и следующим действиям.

Отвечай не как обычный чат-бот, а как деловой помощник для команды.

Правила ответа:
1. Сначала дай короткий вывод в 1–3 предложениях.
2. Затем структурируй ответ по смысловым блокам.
3. Выделяй риски, ограничения и слабые места.
4. В конце предлагай конкретные следующие действия.
5. Если данных недостаточно, не выдумывай. Скажи, каких данных не хватает, и задай до 3 уточняющих вопросов.
6. Не используй лишнюю воду.
7. Пиши ясно, практически и по делу.
8. Отвечай на языке вопроса пользователя.

Формат ответа:
- Краткий вывод
- Разбор
- Риски / ограничения
- Следующие действия

Не возвращай JSON. Возвращай обычный структурированный текст.
```

---

### 9.4 API-ключ

Правила:

- API-ключ нельзя хранить в клиентском коде;
- используй `.env.local`;
- добавь пример `.env.example`;
- если ключ отсутствует, приложение должно показать понятную ошибку, а не падать.

Пример `.env.example`:

```env
OPENAI_API_KEY=your_api_key_here
```

## 10. UX/UI требования

### 10.1 Общий подход

Интерфейс должен выглядеть как мини-SaaS дашборд, а не как тестовая форма.

Желаемый стиль:

- чистый layout;
- карточки;
- хорошая иерархия;
- аккуратные отступы;
- современная типографика;
- понятные состояния;
- визуальное разделение основного сценария и истории.

---

### 10.2 Рекомендуемая структура экрана

Экран можно построить так:

- верхняя часть:
  - название: **TeamPulse AI**;
  - подзаголовок: **Быстрые структурированные ответы для команды**;

- основная зона:
  - слева или сверху: поле вопроса;
  - рядом или снизу: кнопка отправки;
  - центрально: карточка ответа;

- боковая или нижняя зона:
  - история последних 5 запросов;

- дополнительные элементы:
  - кнопка копирования;
  - loading state;
  - error state;
  - empty state.

---

### 10.3 Обязательные UI-состояния

Реализовать:

- empty state до первого вопроса;
- loading state во время запроса;
- success state с ответом;
- error state;
- copied state;
- disabled state кнопки;
- empty state истории.

---

### 10.4 Текстовые элементы интерфейса

Название:

```txt
TeamPulse AI
```

Подзаголовок:

```txt
Мини-дашборд для быстрых командных решений через AI
```

Placeholder:

```txt
Задайте рабочий вопрос: например, как подготовить запуск проекта к понедельнику?
```

Empty state ответа:

```txt
Задайте вопрос — AI соберёт краткий вывод, риски и следующие действия.
```

Empty state истории:

```txt
История появится после первых запросов.
```

Ошибка:

```txt
Не удалось получить ответ. Проверьте подключение, API-ключ или попробуйте ещё раз.
```

## 12. Логика работы

Основной сценарий:

1. Пользователь вводит вопрос.
2. Нажимает **Спросить AI**.
3. UI показывает состояние загрузки.
4. Frontend отправляет вопрос на server route.
5. Server route отправляет запрос в LLM с system prompt.
6. Ответ возвращается на frontend.
7. Ответ отображается в карточке.
8. Вопрос сохраняется в историю.
9. История обрезается до последних 5 запросов.
10. Пользователь может скопировать ответ.

---

## 13. Требования к коду

### 13.1 Общие правила

Код должен быть:

- читаемый;
- типизированный;
- разбитый на компоненты;
- без лишней сложности;
- без дублирования логики;
- с понятными именами переменных;
- без мёртвого кода;
- без захардкоженных API-ключей.

---

### 13.2 TypeScript

Используй типы для:

- запроса к API;
- ответа от API;
- элемента истории;
- состояния ошибки;
- props компонентов.

Пример сущности истории:

```ts
export type HistoryItem = {
  id: string;
  question: string;
  answer?: string;
  createdAt: string;
};
```

---

### 13.3 Компоненты

Минимально желательно выделить:

- `QuestionForm`;
- `AnswerCard`;
- `HistoryList`;
- `CopyButton`;
- `ErrorMessage`.

---

### 13.4 LocalStorage

Историю нужно хранить в `localStorage`.

Правила:

- не хранить больше 5 элементов;
- при ошибке чтения localStorage не ломать приложение;
- проверять, что код не падает при server-side rendering.

---

## 14. API route

Если используется Next.js, создай route:

```txt
POST /api/ask
```

Request body:

```json
{
  "question": "string"
}
```

Response body при успехе:

```json
{
  "answer": "string"
}
```

Response body при ошибке:

```json
{
  "error": "string"
}
```

Требования:

- проверять наличие вопроса;
- проверять наличие API-ключа;
- ловить ошибки LLM;
- возвращать корректные HTTP status codes;
- не отправлять stack trace пользователю.

---

## 15. Что можно добавить для креативности

Если основной функционал готов, можно добавить одно или несколько улучшений:

- быстрые примеры вопросов;
- бейджи структуры ответа: «Вывод», «Риски», «Действия»;
- выбор режима ответа:
  - «Кратко»;
  - «Подробно»;
  - «План действий»;
- сохранение не только вопроса, но и ответа в истории;
- кнопка «Повторить запрос»;
- счётчик символов;
- небольшая подпись: **Powered by structured team prompt**.

Не добавляй креативность в ущерб стабильности.

## 17. Definition of Done

Проект считается готовым, если:

- приложение запускается локально;
- пользователь может задать вопрос;
- AI возвращает структурированный ответ;
- используется system prompt;
- API-ключ не находится в клиентском коде;
- есть история последних 5 запросов;
- есть кнопка копирования ответа;
- есть обработка ошибок;
- есть loading state;
- есть empty state;
- UI выглядит как мини-дашборд;
- код структурирован по компонентам;
- есть `.env.example`;
- есть краткий `README.md` с инструкцией запуска.

## 21. Жёсткие правила для агента

1. Не задавай лишних уточняющих вопросов, если можно сделать разумное допущение.
2. Не делай обычный чат-клон.
3. Не храни API-ключ на клиенте.
4. Не выводи сырой JSON пользователю.
5. Не оставляй приложение без обработки ошибок.
6. Не усложняй архитектуру без необходимости.
7. Не добавляй функции, которые ломают основной сценарий.
8. Сначала сделай рабочий MVP, потом улучшай UI.
9. Все изменения должны быть применены в файлах проекта, а не только описаны в чате.
10. В конце работы проверь запуск, основные сценарии и обнови README.

## Правила безопасного аудита и очистки кода

Когда пользователь просит проверить, привести в порядок, организовать, отрефакторить или провести аудит кодовой базы, используй skill `safe-code-audit`.

Агент должен сначала изучить правила проекта и его структуру, затем подготовить короткий аудит перед редактированием файлов.

Агент не должен читать, открывать, пересказывать, парсить, копировать, печатать, редактировать или делать выводы о содержимом защищённых файлов.

Защищённые файлы и папки:

- `.env`
- `.env.*`
- `*.pem`
- `*.key`
- `*.crt`
- `*.p12`
- `*.pfx`
- `id_rsa`
- `id_ed25519`
- `secrets.*`
- `credentials.*`
- `token.*`
- `*.sqlite`
- `*.db`
- `*.dump`
- `private/`
- `secrets/`
- `.git/`
- `node_modules/`
- `dist/`
- `build/`
- `.next/`
- `.nuxt/`
- `coverage/`
- `.cache/`
- `tmp/`
- `logs/`

Если защищённый файл кажется релевантным, агент должен пропустить его и сообщить путь, не читая содержимое.

Агент должен отдавать предпочтение минимальным и обратимым изменениям и не должен переписывать весь проект, если пользователь явно об этом не попросил.

Перед редактированием агент должен предоставить:

- проверенную область проекта;
- определённый стек;
- основные проблемы;
- предлагаемые изменения;
- файлы, которые планируется редактировать.

После редактирования агент должен предоставить:

- изменённые файлы;
